# Timeline — 设计文档

- **日期**：2026-06-21
- **状态**：已通过 brainstorming 评审，待转入实现计划（writing-plans）
- **一句话**：一个能在同一条时间轴上同时容纳「138 亿年前」和「今天下午」的跨尺度事件记录 PWA。

---

## 1. 概述与目标

Timeline 是一个记录事件的 web app，但不是普通记事本。它的核心特征是**能记录从地质/宇宙级别的大尺度事件，到某个下午发生的小尺度事件，并在同一时间轴上可视化对比**。

设计目标：

- 在一条轴上跨越约 12 个数量级的时间（数十亿年 ↔ 秒）。
- 自然地表达**模糊时间**与**残缺时间**（只有起点或只有终点）。
- 美观、跨设备（电脑 / 平板 / 手机）。
- 数据完全属于用户、可移植、可离线。
- 架构**高度解耦**，方便后续维护与功能增减。

非目标（v1 不做）：多用户 / 账号系统、实时协作、服务端、图片/附件、插件系统。

---

## 2. 用户与使用场景

- **单用户**（作者本人）。不做注册登录、不区分他人。
- 在多台设备上记录与查看，数据需要能在设备间搬运。
- 典型动作：随手记一个事件（精确或模糊）、给它归类打标签、在时间轴上和别的事件比较先后与跨度、按类别/标签/关键词筛选回看。

---

## 3. 架构总览

**纯客户端的本地优先 PWA**：没有后端、没有账号、没有任何网络服务，完全离线可用。

- 数据存在浏览器本地（IndexedDB）。
- 设备间通过**导出 / 导入文件**搬运；**导入即合并**（见 §6），不会覆盖丢数据。
- 桌面端可选用 File System Access API 绑定一个本地文件**自动存盘**；用户若把该文件放进自己的网盘文件夹，即等于免费获得桌面端自动同步——而 app 本身仍不依赖任何服务。
- 作为 PWA 可安装、可离线（service worker 缓存应用外壳）。

**为什么不是托管后端 / 网盘 API / 自托管？** 经过权衡，用户明确选择「纯本地 + 手动导入导出」，以换取数据完全自有、零服务依赖。代价是手机端的设备间同步是手动的（桌面端可自动）。Web 平台的现实约束：浏览器无法自行把文件搬到另一台设备，任何「自动同步」都必须搭载某种传输服务；File System Access API 只在桌面 Chromium 系可用，iOS/手机浏览器不支持——因此手机端采用手动导入导出。

---

## 4. 解耦原则（可维护、可增减功能）

核心是**一条单向依赖规则 + 几个明确接缝**，而不是到处加抽象层（避免过度设计）。

1. **依赖只准向内指**：`domain`（时间 + 模型）不依赖任何东西 → `data`（存储）只依赖 `domain` → `state` 依赖 `domain` 与仓储**接口** → `ui` 依赖 `state`/`domain`，**永不直接碰 Dexie**。换存储、换渲染都不层层波及。
2. **仓储接口（依赖倒置）**：UI/state 只跟 `EventRepository` / `CategoryRepository` / `TagRepository` 接口打交道，Dexie 是接口背后一个可替换实现。将来「升级 CRDT / 接网盘 / 换 SQLite」只换实现，上层零改动。
3. **时间模块是独立、纯净、零依赖的基石**：`domain/time` 不 import 任何 app 内部东西，纯函数 + 全单测；最稳定、最可复用。
4. **视图都是「数据 + 筛选状态」的平等消费者**：时间轴视图与列表视图都从同一组 selector 读数据，彼此互不知道对方存在。新增视图（日历、关系图谱）= 新增一个消费者，别处不动。
5. **每个功能是可摘除的自包含模块**：搜索、关联链接、Markdown、主题、导入导出各自是窄接口的独立模块，在 shell 的装配点接入；删一个功能 = 删模块 + 一行注册。**禁止功能间互相 import**，通信走共享状态/selector 或类型化事件。
6. **渲染与领域解耦**：时间轴渲染器只接收一个由纯投影函数算好的**场景视图模型**来画，不知道 Dexie、也不知道 React 状态形状。Canvas↔WebGL 互换、改皮肤都不碰数据层。
7. **存储细节不外泄**：HLC、LWW、墓碑、合并逻辑全部关在 `data` 层内，UI 只看到干净的领域对象。
8. **克制**：v1 不做通用插件系统、不做全局事件总线；只立上述高价值接缝，让架构*允许*将来扩展，但现在不为想象需求买单（YAGNI）。

## 5. 时间模型（核心）

整个 app 的基石。需覆盖约 138 亿年前 ↔ 现代秒级，支持公元前与模糊精度。

### 5.1 时间点

```
TimePoint = {
  year: number       // 十进制「年」，proleptic Gregorian + 天文纪年
                     //   0 = 公元前 1 年，负数 = 公元前；用于轴定位与粗粒度显示
  precision: Precision
  civil?: CivilTime  // 当 precision ∈ {日,时,分,秒} 时存在，显示与存储的真值
}

Precision = 'Ga' | 'Ma' | 'ka' | 'millennium' | 'century' | 'decade'
          | 'year' | 'month' | 'day' | 'hour' | 'minute' | 'second'

CivilTime = { y, mo, d, h?, mi?, s? }   // 精确日历值
```

- 用 JS `number`（double）存 `year` 足以覆盖全跨度：精度档位决定所需有效位数，**不会同时要极大量级又要秒级**（地质事件不需要秒，现代秒级事件量级小）。
- 日级及更细的精度，**额外存精确日历字段** `civil` 作为显示与存储真值，避免「年→秒」浮点往返漂移。轴定位仍统一用 `year`。

### 5.2 事件的时间

- **事件 = 可选起点 + 可选终点**，至少有一个：
  - 仅起点 → 开放式/点事件；
  - 起 + 止 → 区间事件；
  - 仅终点 → 开放式（截止）事件。
- 每个端点都是一个 `TimePoint`，各自带精度。

### 5.3 时间模块的职责（纯函数，重点单测）

- **解析**：把用户输入解析为 `TimePoint`，支持「约38亿年前 / 公元前3000年 / 1990年代 / 2026年6月 / 2026-06-21 15:30」等。
- **格式化**：按精度渲染显示文本。
- **比较 / 排序**：跨精度可比。
- **投影**：
  - 线性投影（细节轴）：`year → x`（给定可视窗口）。
  - 对数投影（总览条）：以「现在」为基准，按距今时长做对数压缩。
- **跨度与模糊**：根据精度推导一个事件的「模糊跨度」（如 `year` 精度的点覆盖整年、`Ma` 精度覆盖一段误差带），供渲染体现不确定性。

## 6. 数据模型与合并

### 6.1 实体

```
Event = {
  id, title,
  start?: TimePoint, end?: TimePoint,   // 至少一个
  note: string,                          // Markdown
  categoryId: Id | null,                 // 单一归属，指向分类树节点
  tagIds: Id[],                          // 多标签
  links: Id[],                           // 关联的其它事件 id
  createdAt, updatedAt,                  // updatedAt 为 HLC（混合逻辑时钟）
  deleted: boolean                       // 墓碑（软删除）
}

Category = { id, name, parentId: Id | null, color, order }   // 通过 parentId 构成树
Tag      = { id, name, color }
```

### 6.2 合并语义（导入 = 合并，不覆盖）

- 每条记录携带 `updatedAt`（HLC）与 `deleted`。
- 导入时**按 id 逐条 LWW（最后写入者胜）**：取 HLC 较新的版本；若较新版本是删除（墓碑），则保留删除。
- 分类与标签按 id 并集合并，同样走 LWW。
- HLC 用「混合逻辑时钟」而非物理时间，避免不同设备时钟不准导致的错误胜出。
- 因为是单用户，真正的字段级并发冲突（同一记录同一字段在两设备并发改）极罕见，LWW 取最新即可，体验上几乎无感。

## 7. 存储、持久化与同步

- **工作存储**：IndexedDB，经 Dexie 封装；表 `events` / `categories` / `tags`（+ 元信息）。始终本地。
- **导出**：单个 JSON 文件 `{ version, exportedAt, events[], categories[], tags[] }`，**含墓碑**。
- **导入**：选择文件 → 按 §6.2 合并进本地库。
- **桌面增强**：File System Access API 绑定文件句柄，变更去抖后自动写盘；并支持「从文件重载」以拉取外部更改。
- **手机端**：手动导出 / 导入。
- 上述全部封装在 `data` 层的仓储实现内，UI 不感知。

## 8. 视图与交互

### 8.1 时间轴视图（方案 D：总览 + 细节 双轴）

- **总览条**：对数压缩的全尺度轴；所有事件作标记；带可拖动的「取景窗（lens）」；温度梯度（深时暖 → 现在冷）编码时间深度，青色 = 现在/可交互。
- **细节轴**：线性，滚轮缩放、拖动平移（触屏双指缩放）；刻度单位随缩放自适应（Ga→Ma→ka→…→年→月→日→时→分）。区间画成条、点事件画成点；**精度以柔和的跨度 / whisker 体现模糊**；事件用贪心分行避免重叠。
- 拖动总览取景窗 ↔ 细节轴联动放大对应区段。
- 选中事件弹出详情卡（标题、格式化时间、Markdown 备注、分类、标签、关联）。
- **渲染**：Canvas 画轴与密集标记（扛大量事件）；DOM 叠加悬停/选中的事件卡片与标签。渲染器只吃由纯投影函数算好的场景视图模型。

### 8.2 列表视图

可排序（起始时间 / 标题 / 分类 / 更新时间）、可筛选的事件列表；显示格式化时间、分类、标签、备注摘要——便于精确比对时间。

### 8.3 筛选与搜索

- 左侧栏：分类树（选父类含子孙）、标签（AND/OR）、全文搜索（标题 + 备注）。
- **筛选状态在两个视图间共享**（同一组 selector）。

### 8.4 编辑

事件创建/编辑表单，含**智能时间输入**（自由文本解析 + 结构化「精度 + 数值」选择两种方式），可填起点/终点/备注/分类/标签/关联。

## 9. 视觉设计

- **气质**：「Deep Time Instrument」——把跨 12 个数量级还要读得准这件事，翻译成精密仪器 / 历表的语言。
- **签名元素**：对数总览尺做成一把**刻度随时间深度变密、颜色随时间深度变温**的测量尺；可拖动的取景窗像滑动一片透镜。
- **双主题**：白天（冷调制图纸）/ 夜晚（深板岩 + 一点岩层质感）。同一套布局、同一套字体、同一条温度梯度签名；切换昼夜只改光线，不改结构。可切换、记忆选择，首次跟随系统。
- **配色语义**：暖色端 = 深时；青色 = 现在 / 可交互（取景窗、激活态），两套主题语义一致。配色具体值留实现阶段微调。
- **字体系统**：Space Grotesk（标题/纪元名）+ Inter（正文）+ IBM Plex Mono（所有日期、跨度、读数——等宽让数字成列，是仪器感的承重墙）。
- **响应式**：左栏在手机上收成抽屉；触屏支持手势。
- **质量底线**：键盘可达、可见焦点、尊重 reduced-motion。

## 11. 模块结构

```
src/
  domain/
    time/        # 时间模型：类型/解析/格式化/比较/投影（纯函数，零 app 依赖）
    model/       # Event/Category/Tag 类型与不变式
  data/
    repositories/  # EventRepository 等接口
    dexie/         # 接口的 IndexedDB 实现
    sync/          # HLC、LWW 合并、导入导出、文件持久化
  state/         # 筛选、选中、视图、主题（selector / store）
  ui/
    timeline/    # Canvas 渲染器 + 交互（缩放/平移/取景窗）+ DOM 叠加
    list/        # 列表视图
    editor/      # 事件编辑表单 + 智能时间输入
    shell/       # 布局、左栏、顶栏、视图切换、导入导出、主题切换
    features/    # 搜索、关联链接、Markdown 等可摘除功能模块
```

依赖方向：`ui → state → data(接口) → domain`；`domain` 不依赖任何上层。

## 12. 测试策略

- `domain/time` 用 **TDD**，覆盖解析/格式化/比较/投影/跨度的边界（公元前、0 年、极大量级、各精度往返、闰年与月长、对数投影基准）。这是最易出错也最该锁死的部分。
- `data/sync` 的合并逻辑（LWW、墓碑、HLC 排序）单测覆盖。
- 仓储接口可用内存实现做上层测试，免依赖 IndexedDB。

## 14. 主要风险与权衡

- **手机端同步是手动的**：纯本地方案的代价；以数据完全自有 / 零服务依赖换取。桌面端可借 File System Access API + 自有网盘缓解。
- **浮点时间精度**：以「精度档位决定有效位数」+ 细粒度存 `civil` 真值规避；需在时间模块单测中守住边界。
- **大量事件的渲染性能**：Canvas 渲染 + 分行/聚合；总览条对密集区做聚合标记。
- **对数总览的可读性**：靠清晰的刻度标注与温度梯度辅助理解，不要求用户读精确对数距离（精确比对交给细节轴）。
